# A包域名采购流程智能化

## Problem Frame

金鳞A包业务在应用上线前需要完成两项重复性高、耗时长的准备工作：

1. **域名采购与记录**：每次新增应用时，运维人员需要人工前往 GoDaddy 搜索可用域名。采购规则为：域名必须包含「域名采购表」中的应用名字段，且应用名在域名中连续（不拆分），后缀限定为 .com。若精确匹配域名（如 `ovant.com`）价格过高，则在应用名末尾依次添加字母继续检索（如 `ovantan.com`、`ovantao.com`），直至找到最便宜的 .com 域名，记录到域名采购表的域名列。该流程纯人工操作，效率低且容易遗漏。

2. **应用元数据准备**：应用描述、标题后缀、App 分类、关键词、隐私协议、用户协议、金币截图等元数据内容，需要跨团队协作完成。产品侧已使用多维表格生成部分内容，但运维侧仍需手动搬运、格式转换、逐个平台填入并提交。流程不稳定，搬运过程容易出错，且低门槛重复事项消耗大量人力。

当前流程依赖人工串联多个外部系统（GoDaddy、多维表格、各应用商店后台），缺乏自动化衔接，导致应用上线周期被人为拉长。

**域名采购核心链路**：应用名 → 连续域名候选生成（应用名 + 尾字母扩展）→ GoDaddy 比价 → 选出最便宜 .com 域名 → 记录到域名采购表 → 完成后艾特提出需求的人。

## Requirements

### 域名采购自动化（P0）

- **R1. 域名检索规则**：根据「域名采购表」的应用名字段去检索——域名必须包含应用名且应用名连续，后缀为 .com。例如应用名 `ovant`，若 `ovant.com` 太贵，则在后面添加字母检索价格（如 `ovantan.com`），取最便宜的域名。
- **R2. 顶级域限定**：只采购 .com 顶级域。
- **R3. 采购粒度**：一应用一名——每个应用只记录一个最便宜的 .com 域名。
- **R4. 采购动作**：只记录最便宜的顶级域名到域名采购表，不执行实际下单操作。

### 元数据自动化流水线（P1）

- **R5. 多维表格数据自动同步**：对接产品侧已使用的多维表格数据源，自动拉取应用描述、标题后缀、分类、关键词等结构化内容，无需运维手动搬运。
- **R6. 元数据内容自动生成**：对于尚未由产品侧提供的字段（如隐私协议、用户协议模板、金币截图规则），系统基于预设模板或 AI 生成能力自动产出合规内容，并支持人工审核修订。
- **R7. 多平台自动填入与提交**：将审核通过的元数据按各目标平台（App Store、Google Play、华为、小米等）的格式要求自动映射并填入，支持一键提交或批量提交。
- **R8. 流程稳定性与可追溯性**：整条流水线需具备状态追踪能力（待生成 / 生成中 / 待审核 / 已提交 / 提交失败），失败时自动重试并通知责任人，所有操作留痕便于审计。

## Success Criteria

- 域名采购流程从人工串联变为系统自动化，单个应用的域名准备时间从数小时缩短至 10 分钟以内。
- 元数据准备流程实现产品到运维到平台的全链路自动流转，人工搬运环节归零。
- 流水线运行稳定，连续 30 天内无因流程错误导致的上线阻塞。
- 智能体替代评估报告明确给出各环节的替代可行性与风险等级。

## Scope Boundaries

- **只比价、记录，不下单**：本需求仅涉及域名比价与记录，不涉及实际下单、付款操作。
- **域名匹配规则**：域名必须包含应用名且应用名连续（不拆分字符），后缀为 .com；若精确匹配价格过高，可通过末尾添加字母扩展检索。
- **一应用一名**：每个应用只记录一个最便宜的 .com 域名到域名采购表。
- **交接方式**：全部完成后艾特提出需求的人（通知机制）。
- 不改造 GoDaddy 或应用商店平台的自有系统，仅通过其公开 API 或网页采集方式交互。
- 不替代理财/采购审批流程（因为根本不需要走审批）。
- 不涉及应用代码开发或上架后的运营数据监控。
- 多维表格的数据源格式变更由产品侧负责，本系统只按约定 Schema 消费。

## Key Decisions

- **智能体优先评估，API 兜底**：优先评估智能体方案的可行性，对于智能体无法稳定完成的环节（如需要验证码的网页操作），回退到传统 API 或 RPA 方案。
- **产品数据为源，运维审核为闸**：产品侧多维表格作为元数据的唯一来源，系统只读取不写入；所有自动生成的内容在提交前必须经过运维审核节点。
- **渐进交付**：P0 域名采购优先上线验证，P1 元数据流水线在 P0 稳定后接入，避免一次性改造过多环节导致风险不可控。

## Dependencies / Assumptions

- 假设 GoDaddy 提供稳定的域名查询 API 或网页结构，可被程序访问。
- 假设产品侧多维表格支持程序化导出或提供 API 接口。
- 假设各应用商店后台的元数据提交接口或页面结构在一定周期内保持稳定。
- 需确认外部采购方（如财务/行政）是否接受系统生成的采购任务单格式。

### Additional Clarifications (Round 3 — 表格字段分析)

**分析对象：** `26财年巨风做包表.xlsx`（飞书多维表格导出文件）

**表格基本信息：**
- 总行数：456 行数据（不含表头）
- 总列数：71 列（Column A ~ BM）
- Sheet 名：`数据表`

**域名相关现有字段分析：**

| 列 | 字段名 | 实际数据格式 | 说明 |
|---|---|---|---|
| A | 应用名 | 字符串，如 `Rally` | 触发域名比价的输入字段 |
| I | 正式域名(app/h5/im/log) | **换行分隔**的多行子域名，如 `acai.rallyea.com\ncrap.rallyea.com\nache.rallyea.com\nmamey.rallyea.com` | 已选定的正式域名，多子域名分行记录 |
| BL | 顶级域名 | 字符串，如 `laviern.com`、`fynoew.com` | **已采购的最终域名**（非 .com/.net 顶级域，而是完整二级域） |

**关键发现：**
- BL列「顶级域名」存储的是**已采购的完整域名**（如 `laviern.com`），而非顶级域后缀（如 `.com`）。命名与实际内容存在语义偏差。
- I列「正式域名」存储的是从 BL 域名衍生出的**子域名集合**，多个子域名用 `\n`（换行符）分隔。
- **目前没有候选域名字段** — GoDaddy 比价结果（候选域名列表、最低价域名、价格、来源）没有写入位置。

**建议新增字段方案：**

在 I列「正式域名」**之后**（即 J列位置，原 J列「隐私协议」之前）插入以下 4 个新列，原有 J~BM 列依次右移：

| 新增列位置 | 建议字段名 | 数据格式 | 说明 |
|---|---|---|---|
| J（插入后） | 🔍 GoDaddy 候选域名列表 | 换行分隔的多行域名，如 `xxx.com\nyyy.com\nzzz.com` | GoDaddy 返回的所有可用候选域名（按价格排序） |
| K（插入后） | 💰 GoDaddy 最低价域名 | 单个域名字符串，如 `xxx.com` | 价格最低的候选域名 |
| L（插入后） | 💵 GoDaddy 最低价 | 数字（单位：美元），如 `8.99` | 最低价域名的价格 |
| M（插入后） | 🏪 GoDaddy 价格来源 | 字符串，如 `GoDaddy` | 价格数据来源平台 |

**命名风格说明：** 参考现有列命名惯例（如 `☀️AI用户协议`、`✨AI隐私协议`、`🧐AI关键词`），新增字段使用 emoji 前缀 + 中文描述性名称，与同工作表内的 AI 相关字段风格保持一致。

**字段插入影响评估：**
- 原 J~BM 列（共 64 列）需整体右移 4 列
- 建议优先与产品侧确认后再执行结构变更
- 如飞书多维表格不支持动态新增列，需通过「在 BL 列后追加」或新建子表的方式处理
